Finding 1: Kotlin Reflected/Stored XSS (High x2)

Reflected Cross-Site Scripting (XSS) False Positive
CWECWE-79: Improper Neutralization of Input During Web Page Generation Similarity IDs-304960705 (Reflected XSS), related (Stored XSS) Fileandroid/sdui-core/src/main/java/com/nba/sdui/core/data/AblyChannelManager.kt SourceLine 111: conn.inputStream.bufferedReader().readText() SinkLine 155: activeChannels[channelName] = channel (Map.set)

Justification: False Positive - Wrong Technology Context

This is a native Android application (Kotlin/Jetpack Compose), not a web application. XSS (CWE-79) is fundamentally impossible in this context because:

  • No HTML is generated, rendered, or served by this code
  • The "sink" identified (activeChannels[channelName] = channel) is a Kotlin MutableMap storing an Ably SDK channel reference for lifecycle management - it does not write to any output stream, DOM, or response body
  • The "source" (inputStream) reads an Ably JWT token from a trusted internal identity endpoint (identity.nba.com) over HTTPS
  • The token is validated against a strict JWT regex before use: ^[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]*$
  • The Ably SDK consumes the token internally for authentication - it is never reflected to any user-visible surface

Checkmarx misclassification: The static analyzer incorrectly categorized a native mobile SDK integration as web output generation. The .set() operation on a Kotlin Map is not equivalent to writing to an HTTP response or DOM.

Finding 2: Java Unchecked Input for Loop Condition (Medium x17)

Unchecked Input for Loop Condition Mitigated
CWECWE-606: Unchecked Input for Loop Condition Fileserver/src/main/java/com/nba/sdui/versioning/SchemaVersionFilter.java SourceMultiple controller endpoints via SduiRequestContext ctx SinkstripFields() / stripEnumValues() recursive iteration Count17 findings (14 Unchecked_Input_for_Loop_Condition, 3 SSRF)

Justification: Mitigated - Depth Limit + Controlled Input

The findings flag that user-controlled input (schemaVersion from query params) influences a recursive loop over JSON nodes. Multiple mitigations are in place:

  • MAX_RECURSION_DEPTH = 50: Hard limit prevents unbounded recursion regardless of payload structure
  • Input validation: schemaVersion is validated against ^\d+\.\d+$ regex in SduiRequestContext.setSchemaVersion() - rejects malformed values before they reach the filter
  • Server-composed JSON: The loop iterates over the server's own composed response (not user-supplied documents). The user controls only which schema version rules apply, not the structure being traversed
  • Bounded by design: SDUI responses have a known maximum depth (~10 levels). The 50-level limit is a safety net, not a production path
  • Game ID validation: StatsApiClient.validateGameId() enforces ^\d{10}$ pattern on all game ID inputs (addresses 3 SSRF findings)

Risk assessment: Even without mitigations, the worst case is CPU time iterating a bounded JSON tree (server-composed, max ~50 sections x ~10 depth = ~500 nodes). This is a VPN-internal service with 5 authenticated users.

Mitigations Applied:

// SchemaVersionFilter.java - Recursion depth guard
private static final int MAX_RECURSION_DEPTH = 50;

private void stripFields(JsonNode node, Set<String> paths, String path, int depth) {
    if (depth > MAX_RECURSION_DEPTH) {
        log.warn("stripFields: max recursion depth reached at path={}", path);
        return;
    }
    // ... iteration logic
}

// SduiRequestContext.java - Input validation
public void setSchemaVersion(String schemaVersion) {
    if (schemaVersion != null && !schemaVersion.matches("^\\d+\\.\\d+$")) {
        log.warn("Invalid schemaVersion format: {}, defaulting to 1.0", schemaVersion);
        this.schemaVersion = "1.0";
        return;
    }
    this.schemaVersion = schemaVersion;
}

// StatsApiClient.java - Game ID validation (addresses SSRF findings)
private static final Pattern GAME_ID_PATTERN = Pattern.compile("^\\d{10}$");
private void validateGameId(String gameId) {
    if (!GAME_ID_PATTERN.matcher(gameId).matches()) {
        throw new IllegalArgumentException("Invalid game ID format");
    }
}

Finding 4: JavaScript SSRF (Medium x2)

Server-Side Request Forgery (SSRF) False Positive
CWECWE-918: Server-Side Request Forgery Fileweb/server.js Proxy Targetsidentity.nba.com (Ably token), SDUI_SERVER env var (composition API)

Justification: False Positive - Hardcoded Targets, No User Input

The proxy middleware routes are not user-controlled:

  • /ably-token proxies exclusively to https://identity.nba.com/rttoken - target is a hardcoded string literal
  • /api/* proxies to SDUI_SERVER environment variable (defaults to http://localhost:8080) - set at deployment time, not by request input
  • Users cannot influence the proxy destination through request parameters, headers, or body content
  • pathRewrite rules are static string replacements, not dynamic patterns
  • The web client runs behind corporate VPN on sdui-prototype.tools.internal.nba.com

Checkmarx misclassification: The analyzer flagged http-proxy-middleware usage as SSRF because it forwards requests to another server. However, SSRF requires an attacker to control the destination URL, which is not possible here - destinations are compile-time/deploy-time constants.

// Proxy targets are constants, not user input
const SDUI_SERVER = process.env.SDUI_SERVER || 'http://localhost:8080';

// /ably-token -> hardcoded target
app.use('/ably-token', createProxyMiddleware({
  target: 'https://identity.nba.com',  // <-- constant
  pathRewrite: { '^/ably-token': '/rttoken' },
}));

// /api -> environment variable (set at deploy time)
app.use('/api', createProxyMiddleware({
  target: SDUI_SERVER,  // <-- env var, not request input
}));
